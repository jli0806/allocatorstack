## Architecture Decision: Follow the gstack Way

**gstack has zero Python code.** Skills are pure SKILL.md files (markdown instructions
Claude Code follows directly). Shell scripts and compiled binaries handle only what
Claude can't do natively. AllocatorStack follows this pattern.

**What Claude Code does natively (no custom code needed):**
- Reads PDFs (multimodal — can see the document)
- Compares data across documents and identifies discrepancies
- Drafts IC memos (LLM — this is what it does best)
- Cross-references claims against SEC EDGAR data (via Bash/curl)
- Builds comparison tables and completeness dashboards
- Generates on-site questions from gaps and outliers
- Normalizes fee structures with arithmetic

**What needs small helper scripts (Claude can't do natively):**
- Reliable batch PDF text extraction with page numbers (PyMuPDF) → `scripts/extract-pdf.py`
- SEC EDGAR API wrapper with rate limiting and ADV XML parsing → `scripts/query-edgar.py`
- FINRA BrokerCheck API wrapper → `scripts/query-finra.py`
- Excel comparison matrix generation (openpyxl) → `scripts/generate-excel.py`

**What disappears (vs. the original design doc):**
- The entire `lib/src/allocator_ai/` directory (20+ Python files)
- The `bin/allocator` CLI
- The triple-interface pattern (skill → CLI → lib)
- AgentRegistry, WorkflowContext, PipelineConfig, DataAdapter ABC
- The SQLite provenance store (Claude tracks provenance in JSON workspace files)
- Jinja2 rendering engine (Claude drafts directly, or uses templates as guidance)

## Vision

### 10x Check

The current plan: "Drop in DDQ PDFs, get a structured comparison and IC memo draft."

The 10x version: "Drop in DDQ PDFs, and get an interactive due diligence workspace
that remembers everything." Claude Code creates a workspace where every subsequent
skill builds on the previous one's context. The IC memo links to exact PDF pages.
Every data point traces to its source.

### Platonic Ideal

The user feels: "This is how due diligence should have always worked."

You tell Claude what you're looking for. It shows candidates with data normalized.
As DDQs come back, Claude ingests them and builds the comparison incrementally.
It flags discrepancies ("Manager A says AUM is $5B, but their ADV filing shows $3.2B").
The IC memo writes itself as data accumulates. The PM's job shifts from assembly to judgment.

## How Skills Work (gstack pattern)

Each SKILL.md contains:
1. **YAML frontmatter** — name, description, allowed-tools
2. **Workflow instructions** — step-by-step instructions Claude Code follows
3. **Bash blocks** — call helper scripts when needed (PDF extraction, API queries)
4. **Human gates** — Claude pauses for review between pipeline stages

Example flow for `/analyze-ddq`:
```
User invokes /analyze-ddq
        ↓
SKILL.md instructs Claude to:
  1. Run `python scripts/extract-pdf.py workspace/inputs/*.pdf`
     → produces JSON with page-level text for each PDF
  2. Read the ILPA taxonomy (schemas/ilpa-aima-v1.yaml)
  3. For each DDQ, match extracted text against ILPA questions
     (Claude does this natively — it's reading comprehension)
  4. Write structured output to workspace/ddq-output.json
     with provenance (source file, page number, confidence)
  5. Generate completeness dashboard (Markdown table)
  6. Present summary to user → HUMAN GATE
```

**The key insight:** Claude Code reading a DDQ and mapping answers to ILPA questions
IS the analysis. You don't need a Python extraction engine for what the LLM does
natively. The Python script just handles reliable text extraction from PDF → plain text.
Claude does the structured understanding.

## Failure Modes

| Failure | Skill | Behavior |
|---------|-------|----------|
| PDF won't parse (corrupted) | /analyze-ddq | extract-pdf.py skips file, returns error in JSON. Claude reports to user. |
| <30% questions extracted | /analyze-ddq | Claude flags as INCOMPLETE with list of missing categories. |
| SEC EDGAR unreachable | /analyze-ddq, /monitor-adv | query-edgar.py returns error JSON. Claude notes "SEC data unavailable — ADV cross-reference skipped." |
| FINRA unreachable | /screen-managers | query-finra.py returns error JSON. Red flag check marked INCOMPLETE for that source. |
| Non-ILPA DDQ format | /analyze-ddq | Claude attempts best-effort mapping. Flags low-confidence areas. |
| Scanned image PDF | /analyze-ddq | extract-pdf.py returns empty text. Claude flags "OCR required." |
| Litigation data source unavailable | /screen-managers | Red flag check notes "Litigation search not performed — PACER access not configured." |
| No CRD number available | /analyze-ddq | Asks user to provide CRD. If none, ADV cross-reference skipped with explicit note. |
